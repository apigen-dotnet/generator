# GitHub Actions Guidelines

## Action Versions
- Always use the **latest major version** of official GitHub Actions that supports **Node.js 24**.
- Node.js 20 actions are deprecated and will stop working. Never use actions that run on Node.js 20.
- Before using any action, verify it runs on Node 24 by checking its `action.yml` for `using: 'node24'`.

### Recommended actions (Node 24 compatible)

| Purpose | Action | Version |
|---------|--------|---------|
| Checkout | `actions/checkout` | `@v6` |
| Setup .NET | `actions/setup-dotnet` | `@v5` |
| Setup Node | `actions/setup-node` | `@v5` |
| Upload artifact | `actions/upload-artifact` | `@v7` |
| Download artifact | `actions/download-artifact` | `@v7` |
| GitHub Release | `ncipollo/release-action` | `@v1` |
| Cache | `actions/cache` | `@v5` |

### Actions to avoid

| Action | Reason | Replacement |
|--------|--------|-------------|
| `softprops/action-gh-release` | Stuck on Node 20 | `ncipollo/release-action@v1` |
| Any `@v4` of checkout/upload/download | Node 20 | Use latest major version |

## Runner
- Use `ubuntu-latest` unless the project requires a specific OS.
- For macOS builds, use `macos-latest`.

## Workflow Best Practices
- Pin action versions to major version tags (`@v6`), not commit SHAs or minor versions.
- Use `permissions` block to limit token scope to what's needed.
- Avoid unnecessary `env` workarounds like `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24` — use updated actions instead.

## Release Process
- **NEVER tag a release without verifying CI passes first.** Always run `dotnet build` and `dotnet test` locally before tagging.
- **NEVER tag a release without verifying the NUGET_API_KEY secret is valid** on the target repo. Check with `gh secret list --repo <repo>`.
- NuGet packages are published automatically when a `v*` tag is pushed. The release workflow uses `secrets.NUGET_API_KEY`.
- If the NuGet push fails with 403, the API key is invalid/expired or doesn't have permission for that package ID.

### Pre-release checklist (MANDATORY before tagging)

1. **Verify CI is green:** `gh run list --repo apigen-dotnet/<repo> --workflow ci.yml --limit 1`
2. **Verify NuGet secret exists:** `gh secret list --repo apigen-dotnet/<repo>` — must show `NUGET_API_KEY`
3. **Verify build locally:** `dotnet build --configuration Release` — must be 0 warnings, 0 errors
4. **Bump version** in `src/Directory.Build.props` — commit and push
5. **Wait for CI to pass** on the version bump commit
6. **Then tag:** `git tag v<version> && git push origin v<version>`

### What happens when a tag is pushed

1. `release.yml` triggers on `push: tags: ['v*']`
2. It builds, packs, and pushes the `.nupkg` to nuget.org using `NUGET_API_KEY`
3. It creates a GitHub Release with the `.nupkg` as artifact

### What happens when specs/ changes on main

1. `regenerate.yml` triggers on `push: paths: ['specs/**']`
2. It checks out the generator from `apigen-dotnet/generator` main branch
3. It runs the generator against the spec and commits the regenerated code

### What happens on a weekly schedule (Monday 18:00 UTC)

1. `update-spec.yml` triggers on `schedule: cron: '0 18 * * 1'`
2. It reads `specs/upstream.toml` for the upstream URL and target file
3. If `enabled = false`, it stops
4. It downloads the upstream spec to the target file
5. If a `specs/patch-spec.py` exists, it runs the patch script (idempotent)
6. If the spec changed, it creates a PR with the updated spec
